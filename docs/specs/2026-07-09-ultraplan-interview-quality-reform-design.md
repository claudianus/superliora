# UltraPlan 인터뷰 품질 개혁 — 하이브리드 (엔진 강제 + 프롬프트)

**날짜**: 2026-07-09
**범위**: `packages/agent-core/src/agent/plan/ultra-plan-mode.ts`, `injection/plan-mode.ts`, `tools/builtin/collaboration/ask-user.ts`, `tools/builtin/planning/next-phase.ts`, `exit-plan-mode.ts`
**레퍼런스**: [Q00/ouroboros](https://github.com/Q00/ouroboros) — PATH 라우팅, Dialectic Rhythm Guard, Refine/Restate 게이트, safe-defaults
**선행**: 권한 개혁 (`feat/ultraplan-read-permission-reform`) 완료 후 진행

## 문제

UltraPlan 인터뷰는 "사용자가 진정으로 원하는 것을 이끌어내는" 것을 목표로 하지만, 현재 4가지 약점이 있다:

1. **질문 피로도**: 에이전트가 코드로 답할 수 있는 질문(예: "현재 프레임워크 버전이 뭔가요?")도 사용자에게 묻는다. Ouroboros는 PATH 1-4 라우팅으로 사실은 자동 답, 결정만 사용자에게 묻는다.

2. **대화 박동 부재**: 에이전트가 연속으로 자동 답하거나 같은 갭을 반복 질문할 때 사용자가 AI의 가정을 인지하지 못한다. Ouroboros의 Dialectic Rhythm Guard(3연속 비사용자 답 후 강제 PATH 2)가 없다.

3. **의도 일원화 부재**: 사용자 답이 한 줄로 압축되거나 폐쇄 시 재진술 없이 Design으로 넘어간다. Ouroboros는 Refine 게이트(5섹션 구조 보존)와 Restate 게이트(한 문장 재진술 확인)로 의도를 정확히 고정한다.

4. **엔진 결함**:
   - `MAX_INTERVIEW_ROUNDS=8` 초과 시 탈출 경로 없음 ("Design remains blocked" 경고만)
   - `recentUserPromptTexts`가 인터뷰 시작 전 잡담을 증거에 포함
   - 자동생성 Seed(`autoGenerated=true`)가 드리프트 검사 완전 면제

## 목표

- 인터뷰 피로도 감소 — 사실은 에이전트가 자동 답, 결정만 사용자에게
- Dialectic Rhythm Guard로 사용자 인지 유지
- Refine/Restate 게이트로 의도 일원화
- 엔진 결함 3종 수정
- 업그레이드 제안 품질 향상 — lateral thinking을 메인 인터뷰 루프에 연결

## 비목표

- 인터뷰 위상 머신(research→interview→design→...) 구조 변경
- Ambiguity Score 계산 알고리즘 변경 (LLM 기반 스코어링 유지)
- Ouroboros의 3-레인 팬아웃 폐쇄 검증(closer + contrarian + gap_hunter 서브에이전트) — 이는 별도 작업

## 핵심 원칙: 하이브리드 (엔진 강제 + 프롬프트 가이드)

**엔진이 강제하는 것** (신뢰성 필요, 프롬프트만으로는 LLM이 안 지킴):
- 출처(origin) 추적 — 각 인터뷰 답이 어디서 왔는지 기록
- Dialectic Rhythm Guard — 3연속 비사용자 답 감지 + 강제 알림
- MAX_ROUNDS 탈출 — safe-default 진행 옵션
- 증거 필터 — 인터뷰 시작 시점 이후 프롬프트만
- 드리프트 면제 완화

**프롬프트가 가이드하는 것** (유연성 필요, 컨텍스트에 따라 적응):
- PATH 1-4 질문 라우팅 지침
- Refine 구조화 지침 (답을 5섹션으로 보존)
- Restate 게이트 지침 (폐쇄 전 한 문장 재진술)
- 업그레이드 제안 품질 가이드 + lateral thinking 관점 주입

## 설계 상세

### 1. 출처 태그 + 답 추적 (엔진)

#### 1-1. `InterviewRound`에 origin 필드 추가

```typescript
export type InterviewAnswerOrigin = 'user' | 'code' | 'research';

export interface InterviewRound {
  readonly roundNumber: number;
  readonly question: string;
  readonly userResponse: string;
  readonly timestamp: number;
  /** Where the answer came from. 'user' = user decided directly via AskUserQuestion;
   *  'code' = agent answered from codebase evidence; 'research' = external research. */
  readonly origin: InterviewAnswerOrigin;
}
```

#### 1-2. `InterviewState`에 연속 비사용자 답 카운터 추가

```typescript
export interface InterviewState {
  // ...기존 필드...
  /** Number of consecutive rounds where origin was 'code' or 'research' (not 'user').
   *  Reset to 0 whenever a 'user' answer is recorded. Used by Dialectic Rhythm Guard. */
  readonly consecutiveNonUserAnswers: number;
}
```

#### 1-3. `addInterviewRound`에 origin 전달

`addInterviewRound(question, userResponse, origin)` 시그니처 변경. origin에 따라 `consecutiveNonUserAnswers` 업데이트:
- `origin === 'user'` → 카운터 0으로 리셋
- `origin === 'code' | 'research'` → 카운터 +1

`recordInterviewAnswers`도 origin을 받아 전달. 기본값은 `'user'` (AskUserQuestion 경로).

#### 1-4. 신규 도구: `RecordInterviewFinding`

에이전트가 코드/조사로 답을 발견해 사용자에게 묻지 않고 기록할 수 있게 하는 도구. 이것이 PATH 1(auto-answer)과 PATH 3(research)의 엔진 진입점이다.

```typescript
// tools/builtin/planning/record-interview-finding.ts
export class RecordInterviewFindingTool {
  readonly name = 'RecordInterviewFinding';
  // input: { finding: string; origin: 'code' | 'research'; question_answered: string }
  // → planMode.recordUltraInterviewFinding(finding, origin, questionAnswered)
}
```

이 도구는 interview 단계에서만 허용되고, `plan-mode-guard-deny.ts`의 읽기 게이트 통과 후 별도 허용 항목으로 추가된다. `AskUserQuestion`과 함께 인터뷰 단계의 두 번째 "턴 종료" 도구가 된다.

**왜 새 도구인가:** 현재 구조에서 `AskUserQuestion`은 항상 사용자에게 간다. 에이전트가 "코드로 답할 수 있다"고 판단해도 그 답을 기록할 경로가 없다. Ouroboros는 MCP가 질문을 생성하고 메인 세션이 답하는 구조지만, 우리는 단일 에이전트 루프이므로 에이전트가 자발적으로 finding을 기록하는 도구가 필요하다.

### 2. Dialectic Rhythm Guard (엔진 강제)

`formatInterviewReadinessGuide`에서 `consecutiveNonUserAnswers >= 3`이면 강제 알림을 주입:

```typescript
if (state.consecutiveNonUserAnswers >= 3) {
  lines.push(
    '',
    '⚠ RHYTHM GUARD: 3 consecutive findings were answered from code or research, not the user.',
    'Your next turn MUST use AskUserQuestion (PATH 2) to confirm a decision with the user directly.',
    'Do not use RecordInterviewFinding again until the user has answered at least one question.',
  );
}
```

이것은 프롬프트 권고가 아니라 readiness guide의 강제 블록이다 — `NextPhase`가 blocked인 것처럼, 에이전트가 이를 무시하고 진행하기 어렵다. `interview` 단계에서 `RecordInterviewFinding`이 호출되었는데 `consecutiveNonUserAnswers >= 3`이면, `plan-mode-guard-deny.ts`가 이를 deny하고 AskUserQuestion을 유도하는 메시지를 반환한다.

### 3. 인터뷰 프롬프트 재설계 (프롬프트)

`injection/plan-mode.ts`의 `PHASE_INSTRUCTIONS.interview` 재작성. 현재 "expert leader mindset" 4불렛 구조를 유지하면서 PATH/Refine/Restate 지침을 추가.

#### 3-1. PATH 라우팅 섹션 (새로 추가)

```
Question routing — minimize user fatigue, maximize decision quality:
- PATH 1 (auto-answer): If the codebase, config, or manifest already answers
  the question, use RecordInterviewFinding to record it with origin "code".
  Do NOT ask the user.
- PATH 2 (user judgment): Goal, acceptance criteria, trade-offs, business
  logic, visual/scope preferences → ALWAYS use AskUserQuestion. These are
  human decisions.
- PATH 3 (research): External facts (API versions, pricing, compatibility) →
  research first, then use RecordInterviewFinding with origin "research".
  Confirm surprising findings with the user.
- When in doubt → PATH 2. Asking is safer than guessing.
```

#### 3-2. Refine 지침 (새로 추가)

```
When the user gives a free-text answer (via "Other" or open question), do not
compress it to one line. Structure it before the next round:
- Decision: [the core choice]
- Reasoning: [why, 1-2 bullets]
- Constraints: [user-stated limits]
- Out of scope: [what the user deferred]
- Codebase context: [what you verified from code]
This structure feeds the Seed Spec extraction — sloppy compression loses intent.
```

#### 3-3. Restate 게이트 (새로 추가)

```
Before calling NextPhase to Design, restate the agreed goal in one sentence and
confirm with the user via AskUserQuestion:
  "Based on our discussion, the goal is: <one sentence>. If someone read only
  this line, would they arrive at the same outcome you have in mind?"
Options: [Yes, advance to Design] / [Adjust wording] / [Missing scope].
This is the ONLY point where one-line compression is allowed.
```

#### 3-4. 질문 수 통일

프롬프트와 readiness guide의 질문 수 지침을 `"Ask 1-2 focused questions per call"`로 통일. 현재 프롬프트는 "1-3", readiness는 "1개 갭"으로 불일치.

#### 3-5. 업그레이드 제안 + lateral thinking 주입

`formatInterviewReadinessGuide`에서 현재 perspective에 따라 lateral thinking 질문을 주입. `_generateQuestionsForPersona`의 질문 은행을 인터뷰 관점(researcher/simplifier/architect 등)에 매핑하여, readiness guide의 "NEXT TURN" 초점에 "더 나은 방법" 제안 프롬프트를 추가.

예: `simplifier` perspective + `goal` gap → readiness guide에 `"Simplifier lateral: 'What can we remove without breaking it?' Consider a Baseline that cuts 30%+ of components."` 주입.

### 4. 엔진 결함 수정

#### 4-1. MAX_INTERVIEW_ROUNDS 탈출 (safe-default)

`next-phase.ts`의 interview→design 게이트와 `formatInterviewReadinessGuide` 수정:
- 8라운드 초과 + 여전히 NOT READY → readiness guide에 safe-default 옵션 표시:
  ```
  Round cap reached. Two options:
  1. AskUserQuestion: offer the user to advance with conservative defaults for
     remaining gaps (tagged as assumptions in the Seed Spec).
  2. Continue interviewing to close the remaining blockers manually.
  ```
- `next-phase.ts` 게이트 로직: 8라운드 초과 시, 사용자가 "advance with defaults"를 명시적으로 선택한 경우(AskUserQuestion 응답에 `advance_with_defaults: true` 포함)에만 Design 진행을 허용. 남은 갭은 conservative-default 가정으로 채워진 Seed로 기록.

#### 4-2. 증거 필터 (recentUserPromptTexts)

`startInterview` 호출 시 타임스탬프를 기록. `recentUserPromptTexts`가 인터뷰 시작 이후의 사용자 프롬프트만 반환하도록 필터링:

```typescript
private recentUserPromptTexts(): string[] {
  const interviewStartedAt = this._interviewState.startedAtTimestamp ?? 0;
  return (this.agent.context?.history ?? [])
    .filter((message) =>
      message.role === 'user' &&
      isRealUserPromptOrigin(message.origin) &&
      (message.timestamp ?? 0) >= interviewStartedAt
    )
    .slice(-3)
    .map((message) => extractText(message, '\n').trim())
    .filter((text) => text.length > 0);
}
```

`InterviewState`에 `startedAtTimestamp?: number` 추가. `startInterview`에서 `Date.now()`로 설정.

#### 4-3. 드리프트 면제 완화

`exit-plan-mode.ts`의 `validateUltraPlanDrift` 수정:
- `autoGenerated=true`일 때 완전 면제 대신 **관대한 임계값** 적용. `isDriftAcceptable`의 일반 임계값(combined ≤ 0.4) 대신 자동생성 Seed용으로 combined ≤ 0.6 적용.
- 0.6 초과 시 여전히 인터뷰 재개. 0.4~0.6 사이는 경고만 하고 통과.

```typescript
if (autoGenerated) {
  const metrics = await this.agent.planMode.ultraEngine.calculateDrift(plan, []);
  if (metrics.combined <= 0.6) {
    return { ok: true, metrics, warning: 'Auto-generated Seed drift is within tolerant bounds.' };
  }
  // Fall through to reopen interview.
}
```

## 데이터 흐름

```
인터뷰 라운드:
  에이전트가 질문 초점 결정 (pickNextInterviewFocus + perspective)
    → PATH 판단 (프롬프트 가이드):
       PATH 1 (code): RecordInterviewFinding(origin='code') → addInterviewRound(origin='code')
       PATH 2 (user): AskUserQuestion → recordUltraInterviewAnswers → addInterviewRound(origin='user')
       PATH 3 (research): research + RecordInterviewFinding(origin='research')
    → consecutiveNonUserAnswers 업데이트
    → calculateAmbiguityScore (캐시 + monotonic ready)
    → formatInterviewReadinessGuide:
       - blockers 표시
       - Rhythm Guard 체크 (consecutiveNonUserAnswers >= 3 → 강제 알림)
       - lateral thinking 주입
       - 다음 초점 (pickNextInterviewFocus)
    → 다음 라운드 또는 NextPhase

폐쇄:
  readiness READY → Restate 게이트 (프롬프트) → AskUserQuestion 확인
    → NextPhase({ phase: 'design' })
    → 8라운드 초과 시 safe-default 옵션

Exit:
  validateUltraPlanDrift:
    - autoGenerated: 관대한 임계점(0.6) 검사
    - 비-autoGenerated: 일반 임계점(0.4) 검사
```

## 에지 케이스

- **RecordInterviewFinding 남용**: 에이전트가 모든 질문을 자동 답하려 할 수 있음. Rhythm Guard(3연속)가 강제 차단으로 방지. 차단 메시지는 AskUserQuestion 유도.
- **사용자가 Restate를 거부**: "Adjust wording" / "Missing scope" 선택 시 인터뷰 계속. 강제 진입 아님.
- **8라운드 초과 + 사용자가 safe-default 거부**: 계속 인터뷰. 강제 진입 아님. 다만 무한 루프 방지를 위해 추가 라운드마다 safe-default 옵션 재제시.
- **timestamp 없는 과거 메시지**: `message.timestamp ?? 0 >= interviewStartedAt`에서 0이면 인터뷰 시작 전이므로 제외됨. 안전한 방향.
- **RecordInterviewFinding이 interview 외 단계에서 호출**: `plan-mode-guard-deny.ts`가 deny. interview 단계 전용.

## 리스크

- **회귀**: 인터뷰 엔진은 복잡하고 기존 테스트가 많음. origin 필드 추가는 기존 `addInterviewRound` 호출을 깨지 않게 기본값 `'user'` 사용.
- **RecordInterviewFinding 도입**: 새 도구는 에이전트 행동 변화를 유발. 프롬프트가 이를 적절히 가이드하지 않으면 에이전트가 혼란. 단계적 도입 — 먼저 프롬프트로 PATH 라우팅을 가이드하고, 도구는 그 보조 수단.
- **Rhythm Guard 과잉 차단**: 3연속 자동 답이 정당한 경우(예: 전부 매니페스트 기반 기술 질문) 사용자가 불편할 수 있음. 임계값 3은 Ouroboros 검증값이며, 필요시 조정 가능.
